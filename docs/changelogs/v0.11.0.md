# Graywolf v0.11.0 — Remote KISS TNCs as First-Class Channels

This release makes graywolf a peer of Direwolf for the `NCHANNEL` use
case: you can now connect to a remote KISS TNC over TCP (for example a
LoRa APRS digipeater exposing a KISS server, another graywolf instance,
or any device that speaks KISS-over-TCP) and treat that link as a
**full channel** — beacon onto it, digipeat to/from it, iGate from it,
exactly as if it were an audio-backed modem channel.

See the new [Remote KISS TNC handbook page](../handbook/remote-kiss-tnc.html)
for a worked example bridging VHF APRS to a LoRa APRS digipeater.

## New features

### KISS TCP-client interface type

A new `tcp-client` KISS interface type dials a remote `host:port` and
maintains a single persistent KISS connection. The supervisor reconnects
on drop with exponential backoff + jitter (defaults: 1 s initial, 5 min
maximum, 25 % jitter) and surfaces its state in both the web UI and
Prometheus metrics. A new `POST /api/kiss/{id}/reconnect` endpoint lets
operators short-circuit the current backoff wait when they know the peer
is back up.

The existing `tcp` (server-listen) type is unchanged.

### KISS-only channels (no audio device required)

The Channels page gained a segmented type picker at create time:
**Modem-backed** (audio input device required, same as before) or
**KISS-TNC only** (no audio device, no modem — the channel exists only
as a logical routing lane serviced by a KISS TNC interface). The
`channels.input_device_id` column is now nullable; a migration rebuilds
the table on upgrade.

Channel type is read-only on edit, with an explicit **Convert to…**
confirmation flow (delete + recreate semantics) for operators who need
to change it.

### TX dispatch routes per channel backend

Graywolf's TX governor used to hand every frame directly to the modem.
It now goes through a per-channel backend dispatcher that resolves each
frame to its channel's backend — audio modem, KISS-TNC (possibly with
multiple interfaces peering the same logical channel), or neither. The
dispatcher fans out to every registered backend for the frame's channel
and returns success if at least one accepted.

A channel has at most one backend *kind* at a time; attaching an audio
input device to a channel already serviced by a TX-enabled KISS-TNC
interface (or vice versa) is rejected at the API with a 400. Dual-kind
channels are disallowed because doubling a TX frame is almost always a
misconfiguration. Multi-interface KISS-TNC channels (e.g. two remote
TNCs peering the same logical channel) are fine; each interface receives
every TX frame.

### TX-from-governor opt-in flag on KISS TNC interfaces

A new `AllowTxFromGovernor` (`allow_tx_from_governor`) boolean on
TNC-mode KISS interfaces controls whether the TX dispatcher treats the
interface as a backend for its channel.

- **Existing TNC-mode server interfaces** (rows that predate this
  release) **default to `false`**. Upgrading is a no-op: no interface
  starts transmitting frames it wasn't transmitting before. To opt in,
  edit the interface and check the new **Transmit from digipeater /
  beacon / iGate to this interface** box.
- **Newly created tcp-client interfaces** default to `true`, because
  the typical intent of dialing a remote TNC is to make it a full
  bidirectional participant.

### Channel pickers show backing + health everywhere

Every channel picker across the UI (Beacons, Digipeater, iGate, KISS)
now renders each channel with a three-state glyph and text:

- **● Live** — backend configured and up
- **○ Backend down** — backend configured but not currently running
- **— Unbound** — no backend configured

The pickers are backed by a shared `channelsStore` (5 s polling,
revalidate-on-focus) so every page agrees on each channel's current
state. On backing-state transitions while the user is on a page, the
affected row / card briefly pulses.

The pickers themselves moved from native `<select>` elements to a
custom accessible listbox (`role="combobox"` + `role="listbox"`,
arrow-key navigation, typeahead, `aria-activedescendant`). The custom
listbox renders a two-line option layout that native `<select>` could
not handle on narrow screens.

### Digipeater rule-type picker (same-channel vs. bridge)

The Digipeater UI used to hardcode `to_channel = from_channel`. It now
presents an explicit radio group: **Repeat on same channel** (default,
hides the To picker and sets `to = from` on save) or **Bridge to
another channel** (reveals the To picker). When the two channels'
backings differ in kind (audio modem ↔ KISS-TNC), an inline notice
surfaces under the pickers so the operator explicitly sees that frames
are about to cross physical networks.

### Channel delete protects against silent orphans

`DELETE /api/channels/{id}` used to succeed unconditionally and leave
dangling references in beacons, digi rules, iGate config, and KISS
interfaces. It now:

- Returns **409 Conflict** with a structured `{referrers: [...]}` body
  when any row references the channel.
- Performs an atomic per-table cascade when called with `?cascade=true`:
  beacons / digi rules / iGate RF filters / TX timing rows are deleted,
  the iGate singleton's RF and TX channel fields are cleared to null,
  and KISS interfaces pointing at the deleted channel have their channel
  cleared with a `needs_reconfig` flag that surfaces as a yellow banner
  on the KISS page until the operator reassigns a valid channel.

The Channels page exposes a two-step delete dialog: first an impact
review listing every affected row, then a typed-name confirmation gate
(GitHub repo-delete pattern) before the cascade runs.

A new endpoint `GET /api/channels/{id}/referrers` powers the first
dialog. Write-time validation at every soft-FK boundary (Beacon /
DigipeaterRule / KissInterface / IGateConfig / IGateRfFilter /
TxTiming) rejects any create or update that references a nonexistent
channel with a 400.

A one-shot startup scan logs any pre-existing orphan references at
warn level, one line per affected table. It does not delete anything;
use the Channels cascade-delete flow to clean up.

### Observability

New Prometheus metrics:

- `graywolf_kiss_client_connected{interface_id,name}` — gauge, 1 when
  the tcp-client's dialed connection is live.
- `graywolf_kiss_client_reconnects_total{interface_id}` — counter, one
  increment per successful dial since process start.
- `graywolf_kiss_client_backoff_seconds{interface_id}` — gauge, current
  backoff wait in seconds.
- `graywolf_kiss_client_tx_drops_total{interface_id,reason}` — counter,
  TX frames dropped at the per-interface queue (reason: `busy` or
  `down`).
- `graywolf_kiss_instance_tx_queue_depth{interface_id}` — gauge,
  current depth of the per-interface tx queue.
- `graywolf_tx_backend_submits_total{channel,backend,instance,outcome}`
  — counter, TX frames fanned out by the dispatcher. Outcome is `ok` /
  `backend_busy` / `backend_down` / `err`.
- `graywolf_tx_no_backend_total{channel}` — counter, frames submitted
  to a channel with no backend. Should stay at zero; non-zero is
  alert-worthy.
- `graywolf_tx_backend_duration_seconds{channel,backend}` — histogram,
  per-instance Backend.Submit latency.

A ready-to-import Grafana dashboard ships in
`packaging/grafana/remote-kiss-tnc.json` with panels for tcp-client
connection state, reconnect rate, backoff wait, TX drops, queue depth,
and per-channel dispatch outcomes.

Every frame now carries a monotonic `FrameID` propagated through the
governor, dispatcher, each backend, and the wire encoder, so a single
submission can be correlated across the log trail.

### Smoke test

`scripts/smoke/remote-kiss-tnc.sh` spins up a mock KISS TCP server and
drives a graywolf `kiss.Manager` tcp-client end-to-end, asserting a
frame submitted via the per-channel TX dispatcher reaches the remote
peer. Run it locally before releasing to verify the full chain on your
platform.

## Upgrade notes

- **Database migration runs on first boot.** Two new migrations apply:
  one rebuilding `channels` to make `input_device_id` nullable (full
  12-step SQLite table rebuild, wrapped in a transaction), and one
  ALTER TABLE on `kiss_interfaces` to add TCP-client / opt-in /
  needs-reconfig columns. Both are reversible; a prior-release fixture
  test exercises the migration chain in CI.
- **Zero observable TX change for existing deployments.** TNC-mode
  server interfaces that existed before the upgrade continue to behave
  as they did — receiving from KISS clients, not transmitting frames
  the governor schedules. To enable governor TX on an existing TNC
  interface, edit it and check **Transmit from digipeater / beacon /
  iGate to this interface**.
- **Startup orphan warnings are informational.** If graywolf logs
  `orphaned channel references at startup` on first boot after the
  upgrade, that means the database already contained dangling
  references to channels that were deleted before this release shipped
  (the pre-cascade-delete era). Boot continues normally; clean up via
  the Channels cascade-delete UI when convenient.

## Breaking changes

None. The opt-in flag is off by default on migrated rows, so no
existing deployment sees any observable change unless an operator
explicitly enables TX on an interface.
